## 1. 目标

### 完成定义

`/refactor test {path}` 可对任意 SKILL.md 执行结构验证——检查必需 section 存在性、术语一致性、规则无矛盾——输出通过/失败报告和具体修复建议。

### 价值目标

用户可以在修改 skill 后用一条命令验证结构完整性，不再依赖实际使用中出错才发现问题。

### 关键假设

- **假设**: 结构化的验证规则可以在 skill 投入使用前捕获大部分结构性问题（缺失 section、术语不一致、规则矛盾）
- **类型**: 技术可行性
- **验证方式**: 对 ≥2 个真实 skill 执行 test，故意引入 3 种结构问题（缺 section、术语不一致、规则矛盾），验证检出率 ≥80%

## 2. 计划

### 预期交付物

- 代码: `refactor test` 子命令实现
- 文档: SKILL.md 中 test 子命令的验证规则集和工作流定义
- 数据: 验证规则清单（基于 M1 结构规范 + M2 骨架模板）

### 达标标准

| 维度 | 标准 |
|------|------|
| 功能完整性 | 检查 ≥3 类结构问题（缺失 section、术语不一致、规则矛盾），输出通过/失败 + 修复建议 |
| 质量 | 对故意引入的结构问题检出率 ≥80%（经验值: 10 个故意缺陷检出 ≥8 个） |
| 性能与稳定性 | 本里程碑不涉及 |
| 安全 | 本里程碑不涉及 |
| 可维护性 | 验证规则与 M1 refactor 规则集、M2 骨架模板保持一致 |

### 预估投入

1-2 天全职开发

### 决策路径

- **通过**: v1 闭环——创建/测试/重构三环节工具链可用，开始在实际 skill 开发中持续使用验证效果
- **未通过**: 分析未检出的结构问题类型，补充验证规则或调整检测策略

### 风险

依赖 M1 和 M2 产出的结构规范稳定；风险: 验证规则覆盖面不足导致误报/漏报
